          AIDAN'S PERSONAL GIT FORMULA v2
     Selective Add → Commit → Switch Branches → Push Separately
                          Last updated: Feb 2026
================================================================================

QUICK WORKFLOW OVERVIEW

1. See changes & current branch
2. Stage only wanted files
3. Commit
4. Switch to target branch(es) if needed
5. Pull latest (sync)
6. Merge or cherry-pick your work (if switching)
7. Push to each branch separately

DETAILED STEPS

0. Always know where you are (run frequently)

   git branch                # list branches, * = current
   git branch -vv            # shows tracking + ahead/behind
   git status                # changes + current branch

1. Stage ONLY what you want

   git add src/auth.js
   git add tests/ auth.test.js README.md
   git add src/components/            # whole folder
   git add *.css                      # pattern

   # Remove from staging if you added too much
   git reset HEAD path/to/file.txt

2. Check staging area

   git status
   git diff --staged          # preview what will be committed

3. Commit

   git commit -m "Add password reset flow + tests"

   # Better (multi-line)
   git commit -m "Add password reset flow + tests

   - UI form + validation
   - Backend endpoint
   - 4 unit tests"

4. Switch branches (most important new part)

   # List all local branches
   git branch

   # Switch to an existing branch
   git checkout develop
   git checkout main
   git checkout fix
   git checkout feature

   # Modern & safer alternative (Git 2.23+)
   git switch develop
   git switch main
   git switch -c new-hotfix      # create + switch in one command

   # Quick switch back to previous branch
   git checkout -
   git switch -

   # Switch + create branch if it doesn't exist
   git checkout -b feature/payment-v2
   git switch -c fix/bug-127-crash

5. After switching → sync with remote (very important!)

   git fetch origin
   git pull origin develop       # or main / fix / feature
   # or safer (avoid merge commit)
   git pull --rebase origin develop

6. Bring your work to this branch (choose ONE method)

   A. Merge (clean history, creates merge commit)
      git merge feature           # brings all commits from feature

   B. Cherry-pick specific commit(s)
      git cherry-pick abc1234     # one commit
      git cherry-pick abc1234 def5678  # multiple

   C. Rebase your feature onto current branch (linear history)
      (do this while on your feature branch first!)
      git rebase develop
      # then switch back and merge or fast-forward

7. Push to each branch separately

   # Update the branch you're currently on
   git push origin develop

   # First time pushing branch → set tracking
   git push -u origin develop

   # Push from current HEAD to named remote branch
   git push origin HEAD:fix
   git push origin HEAD:main
   git push origin feature:develop     # different local/remote name

   # If rejected (non-fast-forward)
   git push --force-with-lease origin HEAD:main
   # shorter
   git push -f --force-with-lease origin main

EXAMPLE SESSION – Add change → push to feature + develop

# Start on feature
git status
git add src/payment/
git commit -m "Add payment method selector"

# Sync
git fetch origin

# Push to feature (your working branch)
git push origin feature

# Switch to develop and bring the change
git switch develop
git pull --rebase origin develop
git merge feature               # or git cherry-pick <commit-hash>
git push origin develop

# Optional: go to main too
git switch main
git pull --rebase origin main
git merge develop
git push origin main

CHEAT SHEET – Branch switching commands

git branch                     → list
git switch develop             → go to develop
git switch -c fix/bug-456      → create & switch
git checkout -                     → previous branch
git fetch origin                   → update remote info
git branch -d old-branch           → delete local branch
git push origin --delete old-branch → delete remote branch

SAFETY REMINDERS
- Always git fetch / pull before important switches
- Use --force-with-lease (not plain --force)
- Prefer merge or rebase over direct force-pushes to main
- main is usually protected → direct push often blocked
